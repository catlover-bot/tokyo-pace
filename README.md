# TOKYO PACE

## 動的歩行ルーティング

利用者が対象地域内の出発地・目的地を地図またはプリセットで指定し、検索ボタンを押した場合だけ、React → `POST /api/routes` → Cloudflare Worker → openrouteservice Directions APIの順で経路候補を取得します。ブラウザからopenrouteserviceへ直接接続せず、`OPENROUTESERVICE_API_KEY`はWorker Secretだけで使用します。

候補は標準歩行、階段回避を要求した歩行、車いすプロファイルの3種類です。道路ネットワークと属性はOpenStreetMap由来です。車いすプロファイルや階段回避要求は実際の通行可能性、工事、段差、路面状態を保証しません。

## 3経路の比較と推奨理由

検索後は、推奨順位、距離、所要時間、最大連続歩行、最長休憩空白、公衆トイレ候補、給水空白、条件適合、条件負担スコアを比較します。条件負担スコアは低いほど、設定した歩行条件に近い候補です。デスクトップでは比較表、狭い画面では縦並びカードを中心に表示し、条件を満たさない候補も隠しません。

推奨候補は既存の決定的な条件負担スコア（低いほど設定した歩行条件に近い）で1位になった経路です。条件負担スコアの値や計算式は変更せず、同点時だけ、必須条件達成、条件違反数、最大連続歩行、最長休憩空白、所要時間、距離、経路IDの順で決定します。理由文は評価済みの数値から理由コードを純粋関数で生成し、重要度順に最大4件表示します。本番時に生成AIで推薦文を作っていません。

差分の基準は`standard`です。存在しない場合は最短距離、さらに同距離なら経路ID順で決定します。時間差はORSの秒数から計算後に小数1桁、距離・空白は計算後に整数へ丸めます。「標準より328m長い」のように基準を明記します。

比較表・カードの「地図で見る」と地図上の経路クリックは、単一の選択状態で連動します。選択経路は太く、非選択経路は薄く表示し、標準は実線、階段回避要求は破線、車いすプロファイル候補は点線です。選択経路の表示範囲へ地図を合わせます。button、明確なキーボードフォーカス、`aria-current`、表の`caption`と`scope`、文字付きバッジを備え、色だけに依存しません。

条件未達は、連続歩行の上限と超過分、250m以内の公衆トイレ候補、急坂候補、屋内候補を項目別に説明します。確認済み・根拠ありの休憩候補と、自由入館や着席が未確認の`possible`施設を分離し、possibleだけを根拠に休憩可能とは説明しません。条件負担スコアの内訳は所要時間、連続歩行超過、公衆トイレ条件、急坂、屋内条件だけを表示し、現在の計算式にない給水や空白の効果を推測しません。

対象地域は緯度35.67〜35.73、経度139.67〜139.74の新宿駅・東京都庁・新宿中央公園周辺です。外部経路を取得できない場合は、利用者が明示的に`DemoRouteProvider`の固定デモを選べます。比較UIには実APIの経路ではないことを表示し、動的経路と同じ出典として扱いません。

Secretは`wrangler secret put OPENROUTESERVICE_API_KEY`で設定します。ローカルではGit管理されない`.dev.vars`を使用でき、`.dev.vars.example`にはキー名だけを記載しています。値をソース、設定ファイル、ログ、レスポンスへ含めません。

同一要求は丸めた座標、固定プロファイル／options、APIスキーマバージョンから作るキーで15分キャッシュします。APIキーはキーに含めません。距離・時間はopenrouteserviceの推定値で、施設空白と連続歩行は各経路折れ線へ公式候補を再射影して経路距離尺度で計算します。

## 休憩・給水・屋内候補

`npm run data:update`は公式トイレに加え、東京都「だれでも東京」、東京都水道局 Tokyowater Drinking Station、新宿区公共施設情報を更新スクリプトで取得し、文字コード変換・列検証・正規化後のJSONを生成します。ブラウザは外部CSVへアクセスしません。

施設からルートまでは推定直線距離、ルート上の空白は折れ線への射影位置を`DemoRoute.distanceMeters`へ正規化した推定距離です。実際の道路ネットワーク上の徒歩距離ではなく、本番ルーティング導入後は道路区間距離へ置換予定です。

公共施設やバリアフリー情報への掲載は、自由な入館、着席休憩、営業中、座席の空き、安全性を保証しません。給水地点に座席があるとも限りません。休憩地点追加案は最大空白の中点を示す理論上の配置候補です。

## オープンデータ生成の再現性

`data/raw/`の公式CSVは合計約5.8MBで、すべて再配布可能なCC BYです。25MB基準を下回るため、ネットワーク非依存の再生成と監査を目的にraw snapshotをGit管理します。テストfixtureもGit管理します。

取得情報は`data/generated/open-data-manifest.json`へデータセット単位で集約します。各項目は配布URL、取得日時、raw SHA-256、バイト数、正規化・除外件数、上流更新日、文字コード、ライセンスを保持します。レコード本体は`sourceDatasetId`でmanifestを参照し、毎回変化する`retrievedAt`を重複保存しません。同じraw SHA-256では以前の取得日時を維持します。

`data/generated/`は監査・再利用向け全件版、`src/data/generated/`はデモルート近傍だけを含むブラウザ同梱版です。`npm run data:verify-determinism`は保存済みrawから2回生成し、生成JSONと派生CSVのSHA-256一致を検証します。

## 現地確認ワークフロー

現地確認の入力は`data/field-verification/rest-spots.csv`へ保存します。初期ファイルはヘッダーだけで、確認結果を捏造するサンプル行は含みません。列は確認ID、元候補ID、名称・座標・住所、確認日時、確認主体を示す非個人の役割ID、確認方法、一般利用・座席・屋内/屋根・給水・トイレ・車いす対応の三値、観察した営業時間、利用制限、公開可能な根拠参照、注記です。三値は`true` / `false` / `null`を区別し、空欄も`null`として扱って`false`へ変換しません。`verifier`には個人名、連絡先など不要な個人情報を記録しない運用です。

`npm run data:update`は、必須ヘッダー、ID重複、候補ID、座標、タイムゾーン付き日時、三値、確認方法、HTTPSまたは管理対象URNの根拠参照を検証します。行単位の不正は理由コードと行番号を監査へ残して除外し、CSV全体の構造不正や取得失敗時は最初の生成物書込みより前に停止します。全履歴は`data/generated/verified-rest-spots.json`、公開画面で評価に使う縮小版は`src/data/generated/verified-rest-spots.json`へ分離し、公開版に確認者や自由記述を含めません。

`verificationMethod`の許容値は、`on_site_observation`（現地観察）、`combined_on_site_and_official`（現地観察＋公式情報）、`official_source_review`（公式情報だけの確認）、`staff_confirmation`（施設担当者への確認）です。confirmedには前2つ、supportedには`combined_on_site_and_official`だけを使用します。後2つは単独で昇格根拠にせず、再確認時も`verificationId`は重複しない値にします。

休憩信頼度の昇格は決定的な純粋関数で行います。

- `confirmed`: タイムゾーン付き確認日時、確認主体、現地確認方法があり、一般利用可能と座席または明示的な休憩空間を人が肯定確認した地点
- `supported`: confirmedの全条件には届かないものの、公式掲載と有効な現地確認メタデータ・根拠が複数あり、一般利用または座席の肯定情報があり、否定情報がない地点
- `possible`: 公式施設は掲載されているが、自由な入館・着席・休憩可否を確認していない地点、または昇格根拠が不足する地点
- `estimated`: 既存の推定デモ地点

確認日時だけ、公式属性だけ、座席らしい属性だけでは自動昇格しません。厳格な休憩ネットワークと動的経路の`walkingSegments`分割に使うのは`confirmed` / `supported`だけで、`possible` / `estimated`は参考表示に限定します。現地確認が0件なら反映前後の値は同一で、改善値を捏造しません。既存の条件負担スコア、順位、同点処理は変更していません。

現地確認候補は、`data/routing-snapshots/shinjuku-west-to-tocho.v1.json`に保存した新宿駅西口→東京都庁の代表動的3経路（`standard` / `step_avoiding` / `wheelchair_profile`）で順位付けします。このsnapshotは2026-07-23に公開TOKYO PACE Workerから一度だけ取得した正規化済みレスポンスです。OpenStreetMap / openrouteservice由来の保存fixtureであり、最新経路、公式オープンデータ、実測経路ではありません。通常の`data:update`、単体テスト、E2Eはこのファイルだけを読み、外部ORSへ接続しません。固定デモの通常・安心ルートは`fixedDemoRouteMetrics`として回帰比較に残しますが、動的経路の順位点へ混ぜません。

施設のルート投影位置へ直接休憩地点があると仮定した改善を`grossImprovementMeters`とします。施設からルート最近点までの推定直線距離を`estimatedDetourLowerBoundMeters`とし、`max(0, grossImprovementMeters - estimatedDetourLowerBoundMeters)`を`detourAdjustedImprovementMeters`とします。この下限は一方向の直線距離で、実道路上の迂回、往復距離、入口位置を表しません。grossが0以下の候補を除外し、さらに迂回調整後30mかつ現在空白の2.5%以上を満たす候補だけを順位対象にします。30m / 2.5%は簡略化した施設座標と折れ線による微小差を過大評価しない下限です。50m / 5%では境界的な商業候補が除外される感度があるため、30〜49mの候補には「改善は限定的」と明示します。件数を10件へ合わせるための0m候補追加は行いません。

施設アクセス区分は公式データのsource IDと名称から決定的に分類し、利用可能性の確率とは扱いません。公園・屋外公共空間、一般利用目的が明確な公共施設、その他の公式掲載施設、商業施設、ホテル等民間施設、学校・福祉等の順を基本とします。学校・保育・福祉等は、一般公開を推測できず訪問時の負担にも配慮が必要なため順位から除外して監査へ残します。ホテルは休憩可能とみなさず220点のカテゴリ減点と特別注意を付け、それでも大きな迂回調整後改善がある場合だけ上位に残り得ます。

現地確認優先スコアは値の大きい順に訪問優先です。これは利用者向けの条件負担スコア（低いほど条件に近い）とは別の派生分析値です。主経路の迂回調整後改善`×0.6`、改善率`×200`、350mからの近さ`×0.2`、アクセス事前順位、複数動的経路寄与`40点/追加経路`、公式出典品質`12点/段階`を加え、カテゴリ減点と重複施設減点を引きます。同点は迂回調整後改善、改善率、寄与経路数、直線距離、候補IDの順で決定します。理由文は理由コードから固定生成します。

単一の現地確認優先スコアだけで訪問先を決めないため、各候補を3つの迂回仮定で再評価します。`optimistic`は施設が投影位置上にあるとしてgross改善をそのまま使い、`lower_bound`は一方向の施設―ルート推定直線距離を、`conservative_proxy`は単純な往復分としてその2倍を控除します。いずれも`max(0, …)`で、実道路の入口、横断、高低差、通行条件を含む徒歩迂回距離ではありません。往復proxyでも30mかつ2.5%以上残る候補を`robust`、片道控除後が50m未満または5%未満の33〜34m級候補を`marginal`、片道では残るが往復proxyで消える候補を`sensitive`、それ以外を`ineffective`とします。

重み感度は現行の片道控除を固定し、基準1設定と、改善量、改善率、ルート近接、`accessPrior`、`categoryPenalty`、寄与経路数、公式出典品質の7重みを一要因ずつ0.8倍／1.2倍にした14設定、合計15設定で決定的に計算します。基準スコア式自体は変更しません。各候補には基準順位、最良・最悪・平均順位、上位5出現率を保持します。上位5出現率は有限の重み設定に対する順位安定性であり、一般利用できる確率や休憩できる確率ではありません。

さらに、移動改善効果（往復proxy改善、改善率、寄与経路数、ルート距離）と現地確認価値（施設カテゴリ、一般利用目的の明確さ、公式出典品質、現地でしか解消できない不明点、アクセス制限への注意）を分離します。Pareto非劣は、これらの所定指標で他候補に全面的に劣らないという分析上の意味だけを持ちます。最終5地点は、学校・保育・福祉等と`ineffective`を除き、頑健な改善候補、一般利用目的が明確な公共候補、境界仮定を検証する候補を、公開済みの決定規則で構成します。現地確認済みという意味ではなく、結果を見て名称で手選別もしません。

同一施設候補は名称または住所と25m近接を根拠に1地点として数え、レコード数を得点へ加えません。既存confirmed / supported地点との重複、異名称・異住所が同一座標へ集中する上流品質異常、慎重施設、距離・改善基準未達は理由別に監査します。

公開画面の`/?mode=field-check`は読み取り専用の現地調査実施版です。最初に「優先的に確認する5地点」を示し、単一スコア順位は閉じた技術分析へ、今回の優先確認5地点に含めなかった3地点は「その他の分析候補3地点」へ分離します。「現地確認の優先順位」は3つの迂回仮定と15重み設定の分析・確認価値・構成規則による順で、地理的な訪問順や最短巡回順ではありません。候補カードは移動改善効果、現地確認価値、確認優先順位に含めた理由を分け、`robust`等の内部コードも「検討した迂回条件でも改善が残る」「検討した15設定の80%で上位5」のように分析範囲を限定して説明します。

各優先候補カードには、現場で確認する14項目の一時チェック、`verificationId`・`candidateId`・住所のコピー、地図選択を用意します。一時チェックはサーバーや`localStorage`へ保存せず、再読み込みで消えます。Clipboard APIを使えない場合も完全な値を選択できます。地図は代表動的3経路と優先5地点を初期表示し、その他3地点と固定デモは初期非表示です。匿名の書込みAPI、写真アップロード、サーバー保存機能はありません。

全件の重みシナリオと候補集約は`data/generated/field-candidate-ranking-sensitivity.json/.csv`、決定的な5地点は`data/generated/field-visit-shortlist.json/.csv`へ出力します。現地記録用には、確認結果列を空欄にしたUTF-8 BOM付き`data/generated/field-visit-plan.csv`と、ブラウザ同梱用`src/data/generated/field-visit-plan.json`を生成します。全候補テンプレートとは別に5地点CSVを先にダウンロードでき、施設利用可否や調査結果を事前入力しません。`npm run data:verify-determinism`で既存生成物と一緒にバイト単位の再現性を検証します。

## TOKYO PACE分析データ

経路検索後の折りたたみ式「分析データ」から、選択中の経路分析をCSVまたはGeoJSONでダウンロードできます。同一の評価済み経路、manifest、生成時刻から固定列順・固定Feature順で生成するため、同じ入力は同じ内容になります。CSVには経路・profile・距離・所要時間・最大連続歩行・各空白・成立判定・候補数・最大空白の開始/終了・理論追加地点・改善量・manifest参照・警告を含みます。GeoJSONには選択経路、休憩/公衆トイレ/給水の空白、confirmed/supported地点、理論上の追加地点を含めます。

OpenStreetMap由来の経路Featureは© OpenStreetMap contributorsとODbL、東京都・新宿区の地点Featureは各CC BY出典、TOKYO PACEの派生分析は`generatedBy`と生成日時をFeatureごとに保持します。異なるライセンスを単一データセットとして再ライセンスしません。施設までの距離は推定直線、ルート上の空白は経路総距離尺度の推定値で、実際の徒歩経路、安全性、入館・着席・営業状況を保証しません。

「最短ではなく、最後まで歩ける道へ」。TOKYO PACE は、長時間続けて歩くことが難しい高齢者などが、休憩場所・トイレ・坂道を考慮した経路を比較するための Web サービスの MVP です。

> 経路と休憩地点はプロトタイプ用の推定データです。トイレ情報には公式オープンデータを使用しますが、取得後に状況が変わる可能性があり、経路の安全性や設備の利用可否を保証しません。

公開デモ: [https://tokyo-pace.tokyo-pace.workers.dev](https://tokyo-pace.tokyo-pace.workers.dev)

## 解決する課題と対象

一般的な地図の所要時間・距離中心の比較に、最大連続歩行時間、休憩候補、トイレ候補、急坂、屋内休憩候補という判断軸を加えます。主な対象は長時間の連続歩行に不安がある高齢者と、その外出を支援する人です。MVP 対象地域は「新宿駅西口 → 東京都庁」と新宿中央公園周辺に限定しています。

## 実装内容

- 日本語のレスポンシブ UI、キーボードフォーカス、大きな操作領域
- Leaflet + OpenStreetMap の地図（標準は実線、階段回避要求は破線、車いすプロファイル候補は点線）
- 地図外に3経路の比較表、推奨理由、条件違反、詳細カードを表示
- 条件に応じて推奨結果が変わる、説明可能で決定的なスコア計算
- デモ経路の連続歩行区間から、移動継続可能性・最大連続歩行時間・最長休憩空白・上限超過時間を導出
- 公式CSVをビルド前に取得・検証・正規化し、ルート近傍の公式トイレ候補を決定的に評価
- 公式公衆トイレ、車椅子使用者対応情報、推定休憩候補の地図フィルター
- 9,247原レコードを保持した、根拠のある表示候補地点グループ
- 距離帯、近接群、同一座標、空欄率等の品質監査JSON
- 公衆トイレだけを既定条件に使う種別別評価と、デモ折れ線上の最長トイレ空白
- `RouteProvider` 抽象と `DemoRouteProvider` によるルーティング実装の分離
- 3候補の比較中表示、キャンセル、error / retry / 明示的な固定デモ状態
- Vitest の評価ロジック単体テスト
- Cloudflare Workers の API 境界と SPA 静的アセット配信設定

## 技術構成

TypeScript 6、React 19、Vite 8、Leaflet / React Leaflet、Vitest、ESLint、Cloudflare Vite plugin / Wrangler を使用します。Node.js 22 以上（LTS 推奨）と npm が必要です。Cloudflare 公式の [React + Vite ガイド](https://developers.cloudflare.com/workers/framework-guides/web-apps/react/) に沿い、Vite plugin と `wrangler.jsonc` の SPA fallback を採用しています。

表示されたローカル URL（通常 `http://localhost:5173`）をブラウザーで開きます。背景地図の表示には OpenStreetMap タイルへのネットワーク接続が必要ですが、比較情報はローカルデータだけで表示されます。

## 品質チェック

```bash
npm run typecheck
npm run lint
npm test
npm run build
npm run data:verify-determinism
npm run test:e2e
npm run preview
```

`test:e2e`はdevDependencyのPlaywrightとmock Workerレスポンスを使い、実ブラウザで3候補取得、比較、地図選択、詳細開閉、CSV/GeoJSON、エラー後の固定デモ切替を確認します。外部openrouteservice APIは呼びません。Playwrightのブラウザ実体は開発環境のキャッシュへ別途取得され、本番bundleやGit管理対象には入りません。

## 公式オープンデータの更新

```bash
npm run data:update
```

更新処理はブラウザではなく開発時のNode.jsスクリプトで行います。公式CSVを `data/raw/` に保存し、HTTPステータス、文字コード、必須ヘッダー、緯度経度を検証してから、全件版を `data/generated/official-toilets.json`、デモルートから直線距離350m以内の表示用データを `src/data/generated/official-toilets.json` に一時ファイル経由で置換します。失敗時は既存の正常な生成JSONを上書きしません。CSVの列変更で必須ヘッダー検証に失敗した場合は、提供元の最新仕様を確認して正規化処理とfixtureを更新してください。

## Cloudflare Workers へデプロイ

Cloudflare アカウントへログイン済みの環境で、プロジェクト名と `wrangler.jsonc` を確認してから実行します（このリポジトリには認証情報を保存しません）。

```bash
npx wrangler login
npm run deploy
```

`npm run deploy` はビルド後、Worker と SPA の静的アセットを一体で配布します。実運用前に独自ドメイン、ログ保持、セキュリティヘッダー、監視、データ更新手順を設計してください。

## セキュリティ上の注意

API キーや個人情報をコード・デモデータに含めません。`.env` と `.dev.vars` は Git 対象外です。本番化では入力検証、CSP 等の HTTP セキュリティヘッダー、依存関係監査、レート制御、外部 API のタイムアウトと障害時縮退を追加してください。

## 原レコード、候補地点、品質監査

原レコードは削除・統合しません。地図と評価では、同一座標相当の1m以内、または10m以内かつ正規化名称・住所が一致する場合だけ「表示上の候補地点」としてまとめます。これは同一施設との断定ではありません。公衆トイレ、公共施設内の設備情報、鉄道駅内の設備情報はそれぞれ `public_toilet`、`facility_toilet_information`、`station_toilet_information` として区別します。

`data/generated/open-data-audit.json` に、データセット別件数、距離帯、座標範囲、同一座標、近接群、名称・住所一致、空欄率を出力します。`officialToiletRecordCount` は原レコード数、`officialToiletPlaceCount` は表示・評価用の候補地点数です。

## トイレ種別評価と最長トイレ空白

既定のトイレ希望条件を満たすのは、デモルート折れ線から推定直線距離250m以内に `public_toilet` の候補地点がある場合だけです。`facility_toilet_information` と `station_toilet_information` は追加の公式設備情報として件数を表示しますが、一般利用や入場条件を確認できないため、既定条件を成立させません。

最長トイレ空白は、ルート開始点、250m以内にある公衆トイレ候補の折れ線への射影位置、ルート終了点を進行順に並べ、隣接する境界間のデモ折れ線上距離の最大値です。実際の徒歩経路、トイレまでの迂回、利用可否を表す値ではありません。

距離には2つの尺度があります。施設からルートまでの250m判定と最寄り距離は折れ線までの推定直線距離です。一方、ルート進行距離と最長空白は、折れ線上の位置比率をカードの `DemoRoute.distanceMeters` へ換算したルート沿い推定距離です。どちらも実際の道路ネットワーク上の徒歩距離ではありません。本番ルーティング導入後は、正規化値を各道路区間の距離へ置き換える予定です。

## 合理的な仮定と今後の予定

地点座標、経路形状、区間距離、区間歩行時間は UI と導出ロジックを検証する概略値です。公式掲載トイレ候補までの距離はルート折れ線への推定最短直線距離で、道路上の迂回距離ではありません。工事、故障、時間帯、入場制限等で利用できない場合があります。車椅子対応トイレの存在は経路全体の車椅子通行可能性を示しません。次の優先事項は現地検証、データ更新監視、標高による坂道評価、本番ルーティングプロバイダー、ユーザビリティ検証です。詳細は [サービスコンセプト](docs/service-concept.md) と [アーキテクチャ](docs/architecture.md) を参照してください。
